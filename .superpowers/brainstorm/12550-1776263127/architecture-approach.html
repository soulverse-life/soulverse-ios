Architecture Approach

How should the Emotional Bundle feature be structured?

Key Context

The Emotional Bundle is a living safety plan — one per user, 5 editable sections, stored in Firestore. It lives under the Self Care (renamed Tools) tab.

A

Single Firestore Document + Coordinator

Recommended by all experts

  • Data: One document at users/{uid}/emotional_bundle/default (~1.5KB)
  • Architecture: EmotionalBundleCoordinator manages navigation (like MoodCheckInCoordinator)
  • Saves: Per-section field-level updateData — only touches edited section
  • Offline: Firestore SDK handles cache + sync automatically

Pros

  • 1 read per bundle open (cheapest)
  • Atomic per-section saves
  • Follows existing MoodCheckIn pattern
  • Offline-first out of the box

Cons

  • All sections in one doc (but data is tiny ~1.5KB, limit is 1MB)
  • No per-section history/audit trail
B

Subcollection Per Section

5 separate documents under users/{uid}/emotional_bundle/{sectionId}

  • Data: One doc per section (red_flags, support_me, feel_calm, stay_safe, professional_support)
  • Architecture: Same coordinator pattern
  • Saves: Per-document writes

Pros

  • Clean separation per section
  • Could enable per-section listeners

Cons

  • 5 reads per bundle open (5x cost)
  • Over-engineering for tiny data
  • Harder to show completion state (need all 5 docs)
C

Hybrid: Single Doc + Version History Subcollection

Single doc for current state + optional history/{timestamp} subcollection for snapshots

  • Data: Same as A, but saves snapshots on each edit
  • Architecture: Same coordinator + history service

Pros

  • Audit trail for safety-critical data
  • Could show "last edited" per section

Cons

  • YAGNI — no current requirement for history
  • 2x write cost per save
  • Added complexity

Team Recommendation: Option A

All 4 experts converged on the single-document approach. The data is tiny (~1.5KB), the access pattern is simple (fetch once, save per-section), and it perfectly matches the existing MoodCheckInCoordinator + FirestoreMoodCheckInService patterns already in the codebase.

The schema uses a version field for future migration, and arrays are capped at 50 items in security rules (generous headroom beyond the current 2-3 UI slots).
