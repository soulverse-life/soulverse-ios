Why single doc?

Data is tiny (~1.5KB). 1 read per open vs 5 with subcollections. Atomic per-section updates via updateData.

Why arrays with IDs?

Stable identity for client-side diffing. sortOrder makes ordering explicit. Extendable beyond current UI limits.

Why version field?

Enables client-side migration when schema evolves. New sections = new optional array with default [].

Data Flow

Read & Write Patterns
READ (Bundle Open):
Firestore ─getDocument─▶ FirestoreEmotionalBundleService
  ─Result<EmotionalBundleModel?>─▶ MainPresenter
    ─maps to─▶ BundleSectionCardViewModel[] (completion states)
      ─delegate─▶ MainViewController (renders cards)

WRITE (Section Save):
SectionVC ─delegate.didTapSave─▶ Coordinator
  ─service.saveSection─▶ FirestoreEmotionalBundleService
    ─updateData([field: data])─▶ Firestore
      ─on success─▶ pop VC + presenter.refreshAfterSave()

COST:
Open bundle  = 1 read
Save section = 1 write  (field-level merge, only touches edited section)
Offline      = 0 reads (Firestore SDK cache)

Design Decisions Summary

Storage Single doc at users/{uid}/emotional_bundle/default
Section saves Field-level updateData with setData(merge:true) fallback for first write
Offline Firestore SDK handles automatically — last-write-wins per field
Security rules Owner-only read/write, arrays capped at 50 items
Architecture Coordinator + Presenter + ViewModels (follows MoodCheckIn pattern)
Components Reuses SoulverseTextField, SoulverseNavigationView, glass card effects
